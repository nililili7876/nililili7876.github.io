课程目标：

Comprendre les bases d’un système de gestion de version

Savoir utiliser git dans un projet

  • travailler dans un repository local
  • se synchroniser[同步] sur un repository distant
  • avoir une idée des workflows collaboratifs

Gestion de version

1. Git的5大用：

  1. conservation de l’historique
  2. travailler à plusieurs
  3. les branches
  4. travailler avec des contributeurs externes
  5. passage à l’échelle==>扩容

2. 组成:

  1. le repository：Il contient l’historique complet du projet (toutes les révisions depuis le début)

    • Il ne faut pas stocker les fichiers générés par un outil (sous peine de gérer des conflits inutiles)：.o .obj .a .lib .so .dll .class .jar .exe. . .
    • 当脚本由另一个工具生成时，我就会构建它们，例如：生成Makefile的cmake【？？？？？？】

  2. les révisions：Chaque révision

    • introduit des changements par rapport à la révision précédente
    • a un auteur identifié
    • est horodatée：有时间戳
    • contient un message qui décrit les changements

  3. HEAD：révision courante

  4. les tags (étiquettes: 标签)：identifie une révision particulière

  5. les branches

    • est une variante de la collection de fichiers
    • évolue indépendamment des autres branches
    • des opérations de fusion sont possibles

  6. utilisation des branches

    La branche new-feature permet de: développer une nouvelle fonctionnalité et conserver le développement normal

  7. fusion (merge)

tous les changements de la branche sont importés: 所有变化都被导入

Typologie de la gestion de versions: 版本控制的类型化

3. Interactions avec le repository

un référentiel (repository) est une entité opaque[不透明的], pas question de l’éditer directement, donc on doit extraire sa copie de travail (working copy) du repository

checkout: extrait une révision (le plus souvent la dernière) du référentiel,检索版本库的一个（通常是最后一个）修订版

édition: copy到本地随便搞

commit：crée une nouvelle révision à partir de la copie de travail：创建一个新的修订版。Quelques bonnes pratiques de commit

  • Faire des commit souvent
  • Faire des commit relatifs à des changements indépendants dans des
    révisions différentes
  • Ecrire des messages de commit

Utilisation de git en local

  1. Paramétrage à la première utilisation de Git

  2. Création d’un référentiel local

    git init myrepository: 创建

    ls -a myrepository /. git: 显示信息

  3. Premiers commits

    git add file

    git commit [-m message]

    Par défaut les commits sont effectués dans la branche master

  4. Suppression de fichiers

    git rm file
    git commit

  5. Différences entre versions

    git diff [rev_a [rev _b] ]

    • rev_a est l’index
    • rev_b est la copie de travail

    如果你想知道你所处的位置与参照系的关系：git diff HEAD/git diff master

  6. Revenir en arrière 后退

    • git reset
    • git reset —hard
    • git checkout —path

  7. 一些其他的命令

    git status : donne l’état de l’index et de la copie de travail (la liste des modifications prises en compte ou non)
    git show : donne les détail d’un commit
    git log : donne l’historique
    git mv : déplace ou renomme un fichier/répertoire
    git tag : création et suppression des tags

Branches et fusion

1. Branches

Chaque objet de commit possède sa propre liste de commit parents , une branche est juste un pointeur sur le dernier commit.

创建新分支: git checkout -b develop

分支切换用git checkout + 分支名

加-m表示合并到目标分支

2. Fusion de branches

git merge other_branch，没有冲突时git合并的结果会立即提交，新的提交对象有两个父对象，合并历史被记录下来

git merge只适用于自最后一个共同祖先以来在另一个分支所做的修改，如果已经有过一次合并，则只考虑自上次合并以来的变化。

git里面的文件合并

如果同一个文件在两个分支都被修改了，git必须合并他两

  • 文本文件被逐行合并
    • 在单一分支中修改的行会自动合并
    • 如果同一行在两个分支中被修改，git会发送冲突
    • 冲突地区标有<<<<<<< 和 >>>>>>>
  • 二进制文件总是产生冲突
  • 你手动解决冲突（通过编辑文件）
  • git会拒绝提交，直到所有的冲突都被解决。

Travailler à plusieurs sur git

  • 多人合作需要中央服务器，un serveur central
  • 工作方式很像svn
  • 让每个人都能或多或少地在同一时间工作
  • 冲突管理是每个开发者的责任
  • 你需要在服务器上有权限来分配每个开发者
  • 可扩展性不强pas très scalable
